# Ask Mode Integration Design

**Date:** 2026-04-14
**Status:** Approved — ready for implementation planning
**Worktree:** `.claude/worktrees/fix-ai-ask`

## Context

The explore service currently exposes natural-language query ("Ask") as a toggle button squeezed into the navbar next to the classic search input. Clicking Ask swaps the navbar into a narrow input + result strip that renders the LLM's answer as a plain text blob inside the navbar. The feature works end-to-end — entity names in answers are already clickable and deep-link into the Explore pane — but in practice it feels disconnected from the rest of the app:

- The answer surface is a cramped navbar strip, separate from the graph where exploration actually happens.
- Markdown in the LLM response (`**bold**`) is not parsed, so users see literal asterisks around entity names and the entity-link regex fails to match them.
- The feature is the only way Ask touches the rest of the UI: click an entity, it loads into Explore. There is no way for a question to drive Trends, Insights, or Genres, and no way for the graph to mutate directly in response to a question.
- Example queries are hardcoded in HTML, not dynamic.

This spec defines a redesign that makes Ask a true "power-up" layered on top of the existing exploration experience — graph-centric, agent-driven, cross-pane capable, and deliberately *not* a chatbot.

Classic structured search (Artist / Genre / Label / Style dropdown + query box) is untouched. Users pick whichever mental model fits what they're trying to do: structured search for precise lookups, Ask for questions that don't fit that shape.

## Goals

1. Replace the navbar Ask toggle with a **global floating pill** that is visible on every pane, collapses to a small footprint at rest, expands upward into a rich card on click.
2. Make the agent response drive the current view: the LLM returns a list of **structured UI actions** alongside its text answer, and the frontend applies them to reshape the graph / panes / tiles.
3. Surface **dynamic, context-aware suggested queries** (based on the current pane and focused entity) plus a **recent-queries history** in the expanded card — no chat log.
4. Fix the markdown rendering and entity-link regex bug in the summary text.
5. Extract a **shared tool registry** so the NLQ engine and the MCP server both import data tools from one place, preventing drift as new tools are added.

## Non-goals (v1)

- Multi-turn conversation memory. Each Ask is atomic.
- Action-level undo/redo. v1 ships a single "undo last Ask" button that restores the previous view snapshot.
- Touching classic search, OAuth, Discogs sync, or any other navbar element beyond removing the Ask toggle.
- Exposing UI action tools (`seed_graph`, `switch_pane`, etc.) via the MCP server. These are inherently client-specific and stay exclusive to the NLQ response contract.
- Streaming partial UI actions. Actions arrive as a single SSE event after the agent has decided on its plan.

## Product framing

Ask is a co-equal interaction mode, not a replacement for search. The classic structured search stays exactly where it is in the navbar for precise lookups. The Ask pill lives globally and is the way you interact with the music graph through natural language.

Explicitly *not* a chatbot:

- No persistent conversation thread, no bubble UI, no multi-turn memory in v1.
- Each Ask is an atomic action: ask → app transforms → ask again.
- "History" exists only as re-runnable chips in the expanded card.
- The summary strip that appears after a response is a dismissable status bar, not a chat log.

The graph (or whatever pane is relevant to the question) *becomes* the answer. The text summary is supplementary.

## User experience

### Three states of the Ask pill

**Collapsed (resting):**
A small sparkle pill fixed to `bottom: 16px; left: 50%`, visible on every pane. Shows `✨ Ask the graph  ⌘K`. Minimal footprint — never dominates the layout. Keyboard shortcut `⌘K` or `?` (when no input is focused) also opens it.

**Expanded:**
On click or keyboard shortcut, the pill expands upward into a rich card:

- Input row with sparkle icon, autofocus
- **Suggested for you** chip row — dynamic, context-aware queries based on the current pane and focused entity
- **Recent** chip row — last 5 queries from local history
- Click any chip → fills the input and auto-submits
- `Esc` → collapses back to pill

**Post-submit:**
- Card collapses immediately on submit
- Pill shows a subtle loading pulse while SSE streams
- On result: pill flashes `✓ N actions applied` then returns to rest
- A dismissable summary strip docks 60px above the pill showing the text answer + action log
- The graph / active pane has already mutated to reflect the answer
- Click the pill again to ask a follow-up

### Dynamic suggestions

Rendered in the expanded card's "Suggested for you" row. Generated by a new `/api/nlq/suggestions` endpoint that takes the current pane and focused entity (if any) and returns 4-6 ranked queries. v1 implementation is template-based (not LLM-driven) and cached per-context for 5 minutes. Later iterations can call the LLM for smarter suggestions.

### Recent history

Stored in `localStorage` under `nlq.history` (last 5 queries). Rendered as a separate chip row in the expanded card. Silently reset on corruption.

## Architecture

### Four units

#### 1. `common/agent_tools/` — shared data tool registry (NEW)

Pure async data-fetching functions, one per graph query. Zero framework coupling — no FastAPI, no FastMCP, no NLQ engine. Just typed params in, typed dicts out.

**Module layout:**

```
common/
  agent_tools/
    __init__.py          # exports all tool functions and schemas
    schemas.py           # Pydantic models for params and results
    graph.py             # find_path, seed_entities, get_collaborators
    labels.py            # get_releases_by_label_year, get_label_dna
    trends.py            # get_trends, get_graph_stats, get_genre_tree
    entities.py          # get_artist_details, get_label_details, etc.
```

Each function signature is:

```python
async def find_path(
    neo4j: AsyncDriver,
    from_name: str,
    from_type: str,
    to_name: str,
    to_type: str,
    max_depth: int = 6,
) -> PathResult:
    ...
```

Schemas declared as Pydantic models → both consumers auto-derive JSON schema for the LLM tool-calling contract.

#### 2. `api/nlq/` — NLQ engine refactor

- `NLQToolRunner._handle_*` methods become one-line delegates into `common.agent_tools`
- Existing engine loop unchanged; only the tool execution surface swaps
- Agent response extended to `{summary, entities, tools_used, actions}` where `actions` is the list of Tier 3 UI actions the agent decided on
- Streaming SSE events add an `actions` payload event emitted before `result`
- New endpoint `GET /api/nlq/suggestions?pane=...&focus=...` returns dynamic suggestion chips. Added to `api/routers/nlq.py` as a new handler on the existing router (not a separate file).

#### 3. `mcp-server/mcp_server/server.py` — one-line MCP wrappers

Each `@mcp.tool()` function becomes a one-line call into `common.agent_tools`. No behavior change for existing Claude Desktop / external AI users — drop-in refactor verified by regression tests. Any new data tool added for Tier 3 is immediately available to MCP clients too.

UI action tools (`seed_graph`, `switch_pane`, etc.) are *not* exposed via MCP. They have no meaning outside the explore frontend.

#### 4. `explore/static/js/` — frontend Ask surface

- **`nlq-pill.js` (NEW)** — pill state machine (collapsed / expanded / loading / summary), animations, keyboard shortcuts, focus management
- **`nlq-suggestions.js` (NEW)** — fetches from `/api/nlq/suggestions`, renders chip rows, handles click-to-refill-and-submit
- **`nlq-action-applier.js` (NEW)** — receives `actions[]`, validates shape, dispatches each action to the right subsystem (`graph.js`, `trends.js`, `insights.js`, app pane switcher) in a defined order, emits a snapshot for the undo button
- **`nlq.js` (REWRITE)** — thin coordinator between pill, suggestions, and applier; owns SSE connection; builds the summary strip; handles markdown rendering via DOMPurify

#### Surface removed

- `#searchAskToggle`, `#searchModeBtn`, `#askModeBtn` in `explore/static/index.html`
- `#nlqPanel`, `#nlqInput`, `#nlqResult`, `#nlqExamples` and hardcoded example buttons
- `navbar-search-extras` block shrinks or is removed
- The current `nlq.js` entity-link regex approach (replaced by DOMPurify + entity span injection)

Classic search markup and logic (`#searchInput`, `#searchTypeBtn`, `#searchBtn`, `autocomplete.js`, `search.js`) is untouched.

## Action vocabulary (API contract)

The agent returns actions as a JSON array in the SSE `actions` event. Each action has a `type` discriminator and a typed payload. Server validates via Pydantic discriminated union; client validates shape before dispatch and silently skips unknown types (forward-compatible).

```typescript
type Action =
  | { type: "seed_graph"; entities: Array<{name: string; entity_type: "artist"|"label"|"genre"|"style"|"release"}>; replace?: boolean }
  | { type: "highlight_path"; nodes: string[] }
  | { type: "focus_node"; name: string; entity_type: string }
  | { type: "filter_graph"; by: "year"|"genre"|"label"; value: string | number | [number, number] }
  | { type: "find_path"; from: string; to: string; from_type: string; to_type: string }
  | { type: "show_credits"; name: string; entity_type: string }
  | { type: "switch_pane"; pane: "explore"|"trends"|"insights"|"genres"|"credits" }
  | { type: "open_insight_tile"; tile_id: string }
  | { type: "set_trend_range"; from: string; to: string }
  | { type: "suggest_followups"; queries: string[] }
```

### Execution order

The action applier applies actions sequentially in this order regardless of the order in the array (so the agent doesn't have to think about it):

1. `switch_pane` — must happen first so subsequent actions target the right DOM
2. `set_trend_range`, `filter_graph` — scope before mutation
3. `seed_graph`, `find_path`, `show_credits` — content mutations
4. `highlight_path`, `focus_node` — visual emphasis on top of content
5. `open_insight_tile` — drill-down after content is ready
6. `suggest_followups` — stashed in pill state for next expand

### Validation

- Server-side: Pydantic discriminated union. Malformed actions dropped, logged with `⚠️` to `api.nlq` logger.
- Client-side: `nlq-action-applier.js` validates shape. Unknown types logged with `🤷` and skipped.
- String-length caps on all free-text action fields (256 chars).

### Undo model (v1)

No action-level undo. Instead: a single "undo last Ask" button on the summary strip that restores the previous view snapshot. The snapshot is an in-memory object captured by `nlq-action-applier.js` right before the first action is applied, containing: `{ active_pane, graph_nodes, graph_edges, graph_focus, trend_range, open_insight_tile }`. On undo, the applier dispatches the inverse set to the same subsystem handlers. Snapshot is discarded when the summary strip is dismissed or when a new Ask is submitted.

Full action-level undo is deferred.

## Data flow

### On app load

1. `nlq-pill.js` mounts after DOMContentLoaded and checks `/api/nlq/status`. If NLQ is disabled server-side, the pill stays hidden.
2. If enabled, the pill mounts as a collapsed bubble and subscribes to pane-change and graph-focus events so it knows the current context for suggestions.

### Expanding the pill

1. Click (or `⌘K` / `?`) transitions the pill to expanded state. The card animates upward from the pill position. Input receives focus.
2. `nlq-suggestions.js` fires `GET /api/nlq/suggestions?pane=explore&focus=Kraftwerk` in parallel and renders returned chips on completion.
3. Recent queries load synchronously from `localStorage.nlq.history` and render in their own chip row.

### Submitting a query

1. `nlq.js` hits `POST /api/nlq/query` with SSE streaming.
2. SSE events: `status` → `actions` → `result`.
   - `status` events update the card header with a short progress message ("Thinking...", "Querying graph...").
   - `actions` event carries the action list. `nlq-action-applier.js` validates and snapshots the current view, then applies actions in order. Card collapses at this point.
   - `result` event carries the final text summary, entities array, and tools_used. `nlq.js` renders the summary strip above the pill.
3. Pill flashes `✓ N actions applied` then returns to rest.
4. Query is prepended to `localStorage.nlq.history` (capped at 5 entries).

### Dynamic suggestions endpoint (`/api/nlq/suggestions`)

- **Input:** `pane`, `focus` (entity name, optional), `focus_type` (entity type, optional)
- **Output:** `{ suggestions: string[] }` — 4 to 6 ranked query strings
- **Implementation (v1):** template-based. Templates are static strings with `{focus}` substitution, selected by `(pane, focus_type)`. For `pane=explore, focus=Kraftwerk, focus_type=artist` the template set might produce:
  - `Who influenced {focus}?`
  - `What labels has {focus} released on?`
  - `{focus}'s collaborators in the 70s`
  - `Most prolific decade for {focus}`
  - When no focus is set, fall back to a pane-specific default set (e.g., `How are Kraftwerk and Afrika Bambaataa connected?` on Explore).
- **Caching:** 5-minute TTL per `(pane, focus, focus_type)` tuple, keyed in Redis
- **Rate limit:** 100/min per user
- **Failure:** endpoint returning an error causes the expanded card to show only recent history chips — never blocking

### Entity click in summary strip

- Summary text is run through DOMPurify with `ALLOWED_TAGS: ['strong','em','code']`, `ALLOWED_ATTR: []`
- After sanitization, entity spans are injected by walking the text and matching entity names from the `entities[]` array
- Each span is an `<a>` built via `document.createElement`, never `innerHTML`
- Click → switches to Explore pane and loads the entity into the graph (same behavior as today, just built on a correct foundation)

### Error and degraded paths

| Condition | Behavior |
|---|---|
| NLQ service disabled or `/api/nlq/status` fails | Pill stays hidden; classic search unaffected |
| Mid-stream disconnect | Expanded card header shows `⚠️ Connection lost` with retry button |
| Agent timeout (>30s) | Server sends `error` SSE event; client shows toast, collapses pill |
| Individual action fails | Logged with `❌`; other actions still apply; summary strip notes `(N action(s) skipped)` |
| Suggestions endpoint fails | Card shows only recent history chips; no error shown |
| `localStorage.nlq.history` corrupt | Silently reset to `[]` |
| Unknown action type from server | Logged with `🤷`; skipped |

## Security

- **Markdown sanitization** via DOMPurify with tight allowlist. The existing `nlq.js` builds raw text nodes to avoid XSS; the new renderer preserves that guarantee while supporting `**bold**` and `*italic*`.
- **Entity spans** built with `createElement` + `textContent` — never `innerHTML`.
- **Action payload validation** via Pydantic discriminated unions on server and typed validators on client. Unknown fields stripped. String-length caps (256 chars) on all free-text action fields.
- **Rate limiting**: `/api/nlq/query` already has existing middleware limits. `/api/nlq/suggestions` adds a lighter limit (100/min per user).
- **Logging**: query text is hashed for telemetry, never logged verbatim except at `DEBUG` level. No PII in logs.
- **Semgrep** must pass clean on all new code paths.

- `tests/common/test_agent_tools.py` — pytest for each tool function in `common/agent_tools/` against test fixtures
- `tests/api/test_nlq_engine.py` — pytest for engine loop with mocked tools
- `tests/api/test_nlq_actions.py` — pytest for action validation (valid, malformed, unknown type, field-length cap)
- `tests/api/test_nlq_suggestions.py` — pytest for the suggestions endpoint with mocked context
- `explore/__tests__/nlq-pill.test.js` — Vitest for pill state machine (collapsed ↔ expanded ↔ loading ↔ summary)
- `explore/__tests__/nlq-action-applier.test.js` — Vitest for action dispatch with mocked subsystem handlers; verify execution order
- `explore/__tests__/nlq-suggestions.test.js` — Vitest for chip rendering and click-to-submit
- `explore/__tests__/nlq-markdown.test.js` — Vitest for DOMPurify integration, entity span injection, XSS regression cases

### Integration

- `tests/api/test_nlq_query_integration.py` — end-to-end against test Neo4j, asserting `actions[]` shape for canonical queries
- `tests/mcp-server/test_mcp_tools_regression.py` — verify MCP tool wrappers return identical shape after refactor (regression safety for Claude Desktop users)

### E2E (Playwright)

- `tests/e2e/test_ask_pill.py` — open explore, click pill, expand, submit a canned query, verify graph mutates + summary strip appears + pill returns to rest
- `tests/e2e/test_ask_cross_pane.py` — submit a query that triggers `switch_pane(insights)`, verify pane change + tile highlight

### Performance

- Add `nlq_query` and `nlq_suggestions` entries to `tests/perftest/config.yaml` and `tests/perftest/run_perftest.py` per CLAUDE.md rule.

## Telemetry

- Log every dispatched action and every applier success/failure with emojis per `docs/emoji-guide.md`
- Prometheus counter per action type in the API service
- Log suggestion cache hit/miss ratio
- Log `NLQ_HISTORY_RESET` when `localStorage` corruption is detected

## Rollout

- Feature flag behind existing NLQ enable switch — no new flag needed.
- Old `#nlqPanel` / `#searchAskToggle` markup and the entity-link regex path removed in the same PR that ships the new pill.
- One branch, one PR per CLAUDE.md worktree rule.
- Coverage threshold: maintain >80% for all new modules.

## Open questions (none blocking)

None at spec approval time. If the dynamic-suggestions template set proves too thin in practice, we can upgrade v1 to an LLM-driven suggestion call as a fast-follow — the endpoint contract stays the same.
